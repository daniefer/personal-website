Why use Angular?

My number one reason for choosing Angular is personal growth. Since I first started working with Angular way back in 2014, a lot has been added, changed or removed. So keeping up with the latest patterns, hooks and supplemental libraries is quite a task already. Why make it any harder by using some other framework for blogging? Being a full time developer does not mean I get to design pages and web sites from scratch every day. Usually I am leaning on the hard work of others and plugging in new features to existing web apps. Building a blog with Angular lets me stretch those muscles I don't always have the luxury of using ever day.

I already have a website (of sorts) and being able to plug this blog into my existing site without making a bunch of changes is a huge sell for me. You can do this with most any other framework now days, but what makes this particularly interesting to me is how easily I can keep the rest of my site fast by using lazy loading. Not everyone who comes to my fantastic area of the interwebs is going to want to view my ramblings. They may just see my landing page (coming soon) or jump off to my github. In which case why delay! Get there faster, enjoy the ride.

Angular has come a long way since the v1 in terms of tooling. When AngularJs first came on the scene there was little tooling around it. You had to copy from an existing repo template or build the page up from scratch. I am not going to sugar coat it, it was loathsome. I did everything I could to make new features I was building at the time fit into the one working project I had, because starting from scratch was a non-starter. Now days however, we have the angular-cli (queue chorus)! It is easier than ever to get started quickly and be confident that every page will load as long as the build succeeds. My love of build errors over run time error probably stems from my early introduction to compile time languages (C# being my fav'). The angular-cli also added schematics support in v7 which means my creating of new modules is just 4 words away: ng g module blog

Feature Goals

Give all that rational, what are features I want out of this endeavor? Easy. Simple. Cheap.

I want it to be easy to learn what I need/want for the framework when I need it, while at the same not let the plumbing get in the way of actually creating something. This means whatever structure, components, or modules I make it needs to be easy to understand at a glance. I can't be scratching my head for 15 mins once a week/month trying to figure out what steps I am supposed to follow in order to make a new post. I should be open up vscode, issue some new command, and get to writing.

I want it to be simple to create new posts and customize things when I need to. I am aiming for 30 second startup time and automated publishing/deployment. I can't waist time setting up in order to write new posts, spend a bunch of time linking them around, adding photos, or troubleshooting issues with the site. Best to follow KISS here and not get too fancy with building a WYSIWYG editor, databases, or apis calls. I am mostly writing English here, not coding.

I want it to be cheap to host. That means ideally no database, no expensive web servers, no hosting costs. There are a lot of free sites out there already: GitHub Pages, Netlify and even AWS or Azure (if you are willing to relocate at the end of the free trials). Most of these have the restriction of no server side code though. Which means if I want free-free hosting, I cannot be using some custom web server to serve my content. That also means the only time my site can changes is before it is published, cause once it is all the file are static.

Some lesser goals I am wanting to achieve include:

  • Shared styling across all posts.
  • Control of when a post is published.
  • Searchable / SEO

I won't be hitting these last bullets in this post, so stay tuned.
